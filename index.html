🔹 1. Fundamentos y Clasificación de Pruebas
1.1 Clasificación Por Nivel

🔬 Pruebas Unitarias (Componentes)

  • Evalúan la unidad mínima de código (función, método, clase)
  • Ejecutadas por el desarrollador
  • Caja blanca · Aisladas · Rápidas · Automatizadas
  • Se basan en el diseño detallado

🔗 Pruebas de Integración

  • Verifican la interacción entre módulos
  • Detectan fallos de interfaces, contratos y flujos de datos
  • Estrategias: Top-Down, Bottom-Up, Big Bang

🖥️ Pruebas de Sistema

  • Validan el sistema completo en entorno similar al real
  • Flujos end-to-end · Caja negra
  • Las realiza el equipo de QA (no el usuario final)

✅ Pruebas de Aceptación (UAT)

  • Realizadas por usuario / cliente / stakeholders
  • Validan que cumple objetivos de negocio
  • Formales · Caja negra · Previas a producción
1.2 Clasificación Por Objetivo
Tipo¿Qué evalúa?Ejemplos
FuncionalesQué hace el sistema (requisitos funcionales)Login, cálculo de totales, búsqueda
No funcionalesCómo lo hace: rendimiento, seguridad, usabilidadCarga 1000 usuarios, tiempo respuesta <2s
⚠️ TRAMPA FRECUENTE: Las pruebas unitarias NO prueban integración entre módulos ni interfaz de usuario. Son exclusivamente para la lógica interna de una unidad.
🔹 2. Verificación vs. Validación

📋 Verificación — "¿Correctamente?"

  • ¿Estamos construyendo el producto correctamente?
  • Se centra en procesos y artefactos (documentos, código)
  • Técnicas estáticas: revisiones, inspecciones, walkthroughs
  • Valida que el proceso sea el esperado
  • Se aplica en todo el ciclo de vida

✔️ Validación — "¿Correcto?"

  • ¿Estamos construyendo el producto correcto?
  • Se centra en el comportamiento frente a necesidades del usuario
  • Técnicas dinámicas: ejecución de pruebas
  • Valida que el producto sea el esperado
  • Comprueba la idoneidad para el uso esperado
⚠️ REGLA NEMOTÉCNICA:
Verificación → "correctamente" → proceso → estática
Validación → "correcto" → producto → dinámica

🔹 4. Pruebas de Componentes (Unitarias)
Definición y características

Se realizan sobre una unidad de código claramente definida (función, método, clase). El test debe ser:

FIRST Principles

  • Fast — Rápidos de ejecutar
  • Independent — Sin dependencias entre sí
  • Repeatable — Mismo resultado siempre
  • Self-validating — Pasan o fallan, sin ambigüedad
  • Timely — Escritos junto al código

También debe ser

  • Atómico — Prueba la mínima funcionalidad posible
  • Inocuo — No tiene efectos secundarios
  • Automatizable
  • Tests con if/else se separan en 2 casos de prueba
  • Son pruebas de Caja Blanca

¿Quién las realiza?

  • Generalmente el área que construyó el módulo (desarrolladores)
  • Se basan en el diseño detallado
  • Comienzan una vez codificado, compilado y revisado el módulo
¿Qué prueba?
  • La lógica interna del componente (ej. una función)
  • El manejo de errores (valores límite, entradas inválidas)
  • Que se cumpla la especificación del componente
⚠️ NO prueba: integración entre módulos, interfaces de usuario, ni comportamiento del sistema completo.
🔹 5. Pruebas de Integración

Su objetivo es evidenciar defectos en las interfaces e interacciones entre componentes o sistemas integrados. Verifican que partes que funcionan bien aisladas también funcionen en conjunto.

Estrategias de Integración

⬇️ Top-Down

  • Se prueban primero los módulos de mayor nivel
  • Se avanza hacia los módulos inferiores
  • Usa Stubs para simular componentes no disponibles
  • Ventaja: valida la lógica de control desde el inicio

⬆️ Bottom-Up

  • Se prueban primero los módulos de bajo nivel
  • Se avanza hacia módulos superiores
  • Usa Drivers para simular datos de entrada
  • Ventaja: valida componentes fundamentales primero

💥 Big Bang

  • Se acoplan todos los módulos a la vez
  • Reduce cantidad de pruebas
  • Dificulta la detección de errores
  • Requiere todos los módulos probados unitariamente
  • No recomendable para proyectos largos
Integración Incremental — Proceso
  1. Los módulos se van acoplando progresivamente en conjuntos
  2. Tras cada acoplamiento, se prueba la interacción entre módulos
  3. Una vez verificado, se agrega un nuevo módulo y se repiten las pruebas
  4. El proceso se repite hasta completar la aplicación
  5. Cada conjunto debe verificar requisitos funcionales, de rendimiento y seguridad
⚠️ TRAMPA: Stub = simula módulos que no existen (Top-Down). Driver = simula datos de entrada (Bottom-Up). Big Bang ≠ recomendado para detectar errores.
🔹 6. Pruebas de Sistema y Aceptación del Usuario (UAT)

🖥️ Pruebas de Sistema

  • Validan el sistema completo contra los objetivos originales
  • Ambiente de test similar al de producción
  • Ejercita el sistema como lo haría un usuario real
  • Pueden ir de extremo a extremo (end-to-end)
  • Las realiza el equipo de testing del proyecto
  • Son caja negra
  • Incluyen funcionales y no funcionales

✅ UAT – User Acceptance Test

  • Realizadas por usuarios finales / stakeholders / cliente
  • Validan que el sistema se ajusta a sus requerimientos reales
  • Prueba formal con objetivo de aprobación del cliente
  • Se realizan antes de instalar en producción
  • Caja negra
  • Condiciones de prueba basadas en documento de requerimientos
⚠️ TRAMPA CLAVE: Pruebas de Sistema ≠ UAT.
Sistema → ejecutadas por QA del proyecto · UAT → ejecutadas por el usuario/cliente.
No confundir: en Sistema el usuario aún NO valida, eso ocurre en UAT.
🔹 7. Pruebas Complementarias
Pruebas de Rendimiento (Performance)

📈 Carga (Load)

  • Verifica comportamiento bajo el volumen máximo esperado/definido
  • Evalúa capacidad de almacenamiento, procesamiento y cantidad de usuarios
  • Condiciones previstas y normales

💥 Estrés (Stress)

  • Somete al sistema excediendo sus límites de capacidad
  • Considera situaciones NO previstas originalmente
  • Evalúa cómo falla el sistema en el límite

⚡ Rendimiento (Performance)

  • Comprueba la rapidez de respuesta ante una funcionalidad
  • Evalúa tiempos de respuesta, velocidad, escalabilidad
  • Prueba no funcional

⏳ Estabilidad (Soak)

  • Mide rapidez de respuesta trabajando en forma continuada, largos períodos
  • Detecta fugas de memoria, degradación gradual

📊 Escalabilidad

  • Determina la adaptabilidad del sistema
  • Evalúa cómo maneja incremento progresivo de carga
  • Escalado vertical (más recursos 1 server) vs horizontal (más servers)

🔒 Capacidad

  • Define límites máximos: número máximo de usuarios o transacciones
  • Determina cuándo el rendimiento se degrada significativamente
Otras Pruebas Complementarias
TipoDescripción

⚠️ TRAMPAS CLAVE:
• Carga ≠ Estrés (Carga = uso esperado; Estrés = exceder límites)
• Alfa = desarrollador controla el entorno · Beta = usuario controla el entorno
• Smoke Test ≠ pruebas completas (solo verifica que el sistema "arranca")
• Regresión se ejecuta SIEMPRE después de cualquier cambio en el código
🔹 8. Pruebas de Rendimiento con Apache JMeter
8.1 ¿Qué es JMeter?

Herramienta open source para pruebas de carga y rendimiento. Simula múltiples usuarios virtuales y mide el desempeño del sistema. Soporta múltiples protocolos.

8.2 Protocolos Compatibles
HTTP / HTTPS FTP JDBC (Bases de Datos) SOAP / REST LDAP JMS TCP
⚠️ TRAMPA: JMeter NO es solo para web. También prueba bases de datos (JDBC), servicios SOAP/REST, FTP y más.
8.3 Elementos Principales de JMeter

🧵 Thread Group (Grupo de Hilos)

  • Number of Threads: Número de usuarios virtuales
  • Ramp-Up Period: Tiempo (seg.) para lanzar TODOS los hilos
  • Loop Count: Número de repeticiones por hilo
  • Duration: Tiempo total de ejecución
  • Es el punto de partida de todo Test Plan

📡 Samplers

  • Realizan las peticiones al servidor
  • HTTP Request: el más común
  • Configura URL, método (GET/POST), parámetros, body
  • Cada sampler = una petición al sistema bajo prueba

👁️ Listeners (Oyentes)

  • Visualizan y almacenan los resultados
  • View Results Tree: detalle de cada petición
  • Summary Report: resumen estadístico
  • Aggregate Report: percentiles y throughput
  • Pueden guardar resultados en CSV

✅ Assertions (Aserciones)

  • Validan que las respuestas sean correctas
  • Response Assertion: verifica código HTTP (ej. 200 OK)
  • Duration Assertion: tiempo máximo aceptable
  • Size Assertion: tamaño de respuesta
  • Si falla → la muestra se marca como ERROR

⚙️ Config Elements

  • CSV Data Set Config: parametrización con datos externos
  • HTTP Cookie Manager: manejo de cookies de sesión
  • HTTP Header Manager: cabeceras HTTP personalizadas

🔧 Pre/Post Processors

  • Pre-Processor: se ejecuta antes del sampler
  • Post-Processor: extrae datos de la respuesta
  • Regular Expression Extractor: extrae valores con regex
8.4 Parámetros del Thread Group – Detalle
ParámetroDescripciónEjemplo
Number of ThreadsUsuarios virtuales concurrentes100 usuarios
Ramp-Up PeriodTiempo en segundos para lanzar TODOS los hilos (no es la duración total)10s → 10 usuarios/segundo
Loop CountCuántas veces repite cada hilo5 = cada usuario hace 5 requests
DurationTiempo total de ejecución (si Infinite = true)300s = 5 minutos
Startup DelayRetraso antes de iniciar el grupo30s para esperar que otro grupo termine
⚠️ TRAMPA: Ramp-Up ≠ duración total. Si Ramp-Up = 10s y hay 100 threads, se lanza 1 hilo cada 0.1s hasta completar los 100.
8.5 Métodos HTTP: GET vs POST

GET

  • Obtiene/lee datos del servidor
  • Los parámetros van en la URL (query string)
  • Visible en el historial del navegador
  • Puede ser cacheado
  • Menos seguro para datos sensibles
  • Longitud limitada
  • Ejemplo: GET /productos?id=123

POST

  • Envía datos al servidor (crea/modifica recursos)
  • Los parámetros van en el body de la petición
  • No visible en URL · No cacheado
  • Más seguro para datos sensibles
  • Sin límite de tamaño
  • Usado para formularios, JSON payloads
  • Ejemplo: POST /usuarios + body JSON
8.6 Generación de Reporte HTML
  1. Configurar el Listener View Results Tree con un archivo CSV de salida
  2. Ejecutar la prueba para generar el CSV
  3. Ir a Tools → Generate HTML Report
  4. Indicar: archivo CSV resultante, archivo user.properties, directorio de salida (vacío)
8.7 CSV Data Set Config

Permite parametrizar las pruebas con datos reales o generados. Se integran datos de herramientas como Mockaroo, Generatedata.com o Randat.com en formato CSV, y JMeter los lee dinámicamente durante la ejecución.

🔹 10. Métrica APDEX

Application Performance Index. Mide la satisfacción del usuario en función al rendimiento de la aplicación. Valor entre 0 (peor) y 1 (mejor).

APDEX = (NS + NT/2) / N
NS = Solicitudes Satisfactorias (tiempo ≤ umbral satisfactorio)
NT = Solicitudes Tolerables (tiempo entre umbral satisfactorio y umbral de frustración)
N = Total de solicitudes
Umbrales
UmbralDescripciónValor típico por defecto
Toleration ThresholdTiempo máximo tolerable antes de sentir incomodidadConfigurable en jmeter.properties
Frustration ThresholdTiempo que provoca experiencia frustrante (4× el umbral tolerable)Configurable en jmeter.properties
Escala de Valores APDEX
0.94–1.00
Excelente
0.85–0.94
Buena
0.70–0.85
Aceptable
0.50–0.70
Insatisfactoria
<0.50
Inaceptable
⚠️ TRAMPA: APDEX = 1 es la mejor experiencia posible. APDEX = 0 es la peor. El índice 0.94 es el límite entre "Buena" y "Excelente". Valores < 0.50 son inaceptables.
Ejemplo de Cálculo
N = 1000 solicitudes totales NS = 600 satisfactorias (tiempo < umbral tolerable) NT = 200 tolerables (tiempo entre umbrales) APDEX = (600 + 200/2) / 1000 APDEX = (600 + 100) / 1000 APDEX = 700 / 1000 APDEX = 0.70 → Aceptable
🔹 11. Introducción a Selenium

Selenium es una suite open source de bibliotecas y herramientas para la automatización de navegadores web. Es el proyecto paraguas de tres componentes principales.

11.1 Componentes Principales

🚗 Selenium WebDriver

  • Biblioteca para controlar navegadores automáticamente
  • API multiplataforma en múltiples lenguajes
  • Lenguajes oficiales: Java, JavaScript, Python, Ruby, C#
  • Controla navegadores locales y remotos
  • Requiere drivers específicos por navegador

🌐 Selenium Grid

  • Grupo de hosts en red que provee infraestructura de navegadores
  • Permite ejecución paralela de scripts WebDriver
  • Soporta múltiples navegadores, versiones y sistemas operativos
  • Ideal para pruebas de compatibilidad a escala
  • Iniciado por Philippe Hanrigou en 2008

🎬 Selenium IDE

  • Herramienta de Record and Playback (R&P)
  • Record: captura interacciones del usuario como comandos Selenium
  • Playback: ejecuta el script grabado automáticamente
  • Creado por Shinya Kasatani en 2006
  • Extensión de navegador

🔹 14. XPath – XML Path Language

XPath (XML Path Language) es una sintaxis para encontrar cualquier elemento en una página web usando la estructura HTML del DOM. Permite navegación bidireccional (padre-hijo e hijo-padre).

14.1 Sintaxis Estándar XPath
XPath = //tagname[@attribute='value']
// = selecciona el nodo actual (busca en todo el documento)  |  tagname = nombre de la etiqueta  |  @ = selecciona atributo  |  attribute = nombre del atributo  |  value = valor del atributo
14.2 XPath Absoluto vs. XPath Relativo

⚠️ XPath Absoluto

  • Comienza con / (barra simple)
  • Navega desde el nodo raíz (<html>)
  • Ruta completa y directa al elemento
  • Frágil: si la estructura HTML cambia, falla
  • Difícil de mantener
/html[1]/body[1]/section[2]/div[1]/div[2]/a[1]

✅ XPath Relativo (Recomendado)

  • Comienza con // (doble barra)
  • Busca en cualquier parte del documento
  • Más robusto ante cambios de estructura
  • Recomendado para automatización
  • Permite operadores y funciones avanzadas
//input[@name='myName'] //*[@name='myName']
14.3 Operadores Lógicos en XPath

OR — Alguna condición

Se cumple alguna de las condiciones definidas o ambas.

//*[@src='/images/mail.png' or @xpath='1']

Encuentra elementos que tengan src='/images/mail.png' O que tengan xpath='1'

AND — Ambas condiciones

Se deben cumplir ambas condiciones definidas.

//*[@class='social-icon' and @src='/images/twitter.png']

Encuentra elementos que tengan AMBOS atributos con esos valores

NOT — Negación

Excluye los elementos que cumplan la condición.

//*[@type='submit' and not(@id='my-button')]

Todos los botones submit excepto el que tenga ese id

14.4 Funciones Avanzadas de XPath

contains() — Contiene (parcial)

Busca elementos cuyo atributo contiene un valor parcial. Útil para atributos dinámicos.

//*[contains(@id, 'message')] //*[contains(@href, 'github')] //*[contains(@class, 'btn')]

Encontrará "message-error", "message-success", etc.

starts-with() — Empieza con

Busca elementos cuyo atributo comienza con un valor. Ideal cuando los IDs cambian al final.

//*[starts-with(@id, 'user')] //a[starts-with(@href, 'https')]

Encontrará "user-123", "user-456", etc.

text() — Texto exacto

Encuentra el elemento cuyo texto visible coincide exactamente con el indicado.

//*[text()='Historia'] //td[text()='UserID'] //button[text()='Enviar']

Coincidencia exacta del texto visible en página

15.4 Diferencias CSS Selector vs XPath ⚠️ CLAVE

CSS Selector

  • ✅ Más rápido en performance
  • ✅ Más legible y simple
  • ✅ Usado con más frecuencia
  • ✅ Directamente soportado por navegadores
  • ❌ Solo navegación descendente (padre → hijo)
  • ❌ No puede navegar al padre o elemento previo
  • ❌ No puede localizar por texto visible (text() no existe)
  • ❌ Menos flexible para estructuras complejas

XPath

  • ✅ Más potente y flexible
  • ✅ Navegación bidireccional (padre ↕ hijo)
  • ✅ Puede localizar por texto visible (text())
  • ✅ Funciones avanzadas: contains(), starts-with()
  • ✅ Puede seleccionar por posición relativa
  • ❌ Más lento en performance
  • ❌ Más verboso y complejo de escribir
  • ❌ Más difícil de leer para mantenimiento

🔥 Resumen de TRAMPAS y Puntos Clave para el Examen
Verificación vs Validación
Verificación = "correctamente" = proceso = estática | Validación = "correcto" = producto = dinámica
V-Model: Correspondencias
Requisitos ↔ UAT | Diseño Sistema ↔ Pruebas Sistema | Arquitectura ↔ Integración | Componentes ↔ Unitarias
Carga vs Estrés
Carga = uso esperado/definido (dentro de límites) | Estrés = excede límites (situaciones no previstas)
Alpha vs Beta
Alfa = usuario en instalaciones del DESARROLLADOR | Beta = usuario en sus PROPIAS instalaciones
Sistema vs UAT
Sistema = equipo de QA | UAT = usuario final / cliente / stakeholder
JMeter Ramp-Up
Ramp-Up ≠ duración total. Es el tiempo para LANZAR todos los hilos, no el tiempo que corren.
Throughput vs Response Time
Throughput = peticiones/segundo (capacidad) | Response Time = duración de una petición (rapidez)
APDEX Fórmula
APDEX = (NS + NT/2) / N → Excelente ≥ 0.94 | Buena 0.85-0.94 | Inaceptable <0.50
Stub vs Driver
Stub = simula módulos no disponibles (Top-Down) | Driver = simula datos de entrada (Bottom-Up)
XPath Absoluto vs Relativo
Absoluto = empieza con / (frágil) | Relativo = empieza con // (recomendado, robusto)
CSS vs XPath — Navegación
CSS = solo desciende (padre→hijo) | XPath = bidireccional (puede subir al padre)
GET vs POST
GET = parámetros en URL (visible, cacheable, menos seguro) | POST = parámetros en body (más seguro, sin límite de tamaño)
Smoke Test
No es prueba completa. Solo verifica que el sistema "arranca" y sus funciones básicas funcionan.
Pruebas Unitarias
Caja blanca · realiza el desarrollador · NO prueban integración ni UI · basadas en diseño detallado.
